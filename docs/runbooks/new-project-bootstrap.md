# Runbook: bootstrap a new greenfield project

**Audience:** operator with an idea for a brand-new product/service that should be developed via the parallel-dev pipeline.

**Prerequisites:**
- Epic [IFRNLLEI01PRD-910](https://youtrack.example.net/issue/IFRNLLEI01PRD-910) (state refactor) merged
- Epic [IFRNLLEI01PRD-922](https://youtrack.example.net/issue/IFRNLLEI01PRD-922) (parallel-dev) merged + activated
- bootstrap-pack present at `bootstrap-pack/` in claude-gateway (or its own GitLab project once split)

**Total time:** ~1 day for the bootstrap, then minutes per onboarded epic.

---

## Day 0: from idea to spec

### 1. Create the project's git repo

```bash
cd /app/products/
git init <slug>          # or git clone if it exists already
cd <slug>
git commit --allow-empty -m "Initial commit"
```

### 2. Run /bootstrap

Start a fresh Claude Code session in the new repo's directory. Type:

```
/bootstrap
```

This invokes the meta-skill that chains `/specify` → `/constitute` → `/plan` → `/tasks` → validator.

You'll be interviewed via AskUserQuestion to elicit:
- **Specify phase:** target users, success metrics, scope boundaries, integrations, data model, failure modes, non-functional requirements
- **Constitute phase:** which project-specific non-negotiables to add beyond the 7 default articles
- **Plan phase:** architecture decisions (DB, auth, message bus, deployment), each captured as an ADR
- **Tasks phase:** decomposition into ≤4 parallel work_units with files_owned + dependencies + acceptance_test

Iteratively. The validator gates each phase.

### 3. Phase F gate

Once `/bootstrap` finishes, run the final gate:

```bash
bootstrap-pack/scripts/validate-project-spec.py .
```

Must report `17/17 checks passed`. If any check fails, see [spec-lint-failures.md](spec-lint-failures.md) for remediation.

The artifacts you should now have:

```
<project>/
├── PROJECT.md + PROJECT.json
├── constitution.md
├── steering/{coding-standards,security-baseline,test-strategy,repo-conventions}.md
├── adr/0001-record-architecture-decisions.md (at minimum)
├── spec/001-<feature>/
│   ├── requirements.md (EARS-only, REQ-NNN-tagged)
│   ├── design.md + data-model.md
│   ├── contracts/openapi.yaml + asyncapi.yaml + schemas/*.json
│   ├── acceptance/*.feature
│   ├── tasks.json
│   └── quickstart.md
└── .agentic/
    ├── slot-config.entry.json (generated by /bootstrap)
    ├── youtrack-project.json
    └── matrix-room.json
```

## Day 1: onboard to the gateway

### 4. Run project-onboard.sh

```bash
bootstrap-pack/scripts/project-onboard.sh /app/products/<slug>
```

This:
- Re-runs the Phase F validator (defense-in-depth)
- Appends the new slot to `/home/app-user/gateway-state/slot-config.json`
- Prints next-step instructions:
  - Edit the Derive Slot dict in both n8n workflows (Runner + Matrix Bridge) per `docs/runbooks/n8n-code-node-safety.md`
  - Create the YouTrack project with shortName=PREFIX
  - Invite `@claude` to the Matrix room
  - Smoke-test with a trivial `PREFIX-9999` issue

### 5. Edit n8n Derive Slot

Per the printed instructions, follow `docs/runbooks/n8n-code-node-safety.md` to add 2 lines to each Derive Slot Code node:

```js
       : (prefix === 'NEWAPP') ? 'newapp'  // add this line in the if/else chain
```

```js
  'newapp':  { cwd: '/home/.../products/newapp', room: '!RoomID:matrix...' },  // add this line in slotConfig dict
```

### 6. Create YouTrack project + Matrix room

```bash
curl -X POST -H "Authorization: Bearer $YT_TOKEN" -H "Content-Type: application/json" \
  https://youtrack.example.net/api/admin/projects \
  -d '{"shortName":"NEWAPP","name":"NewApp — short description"}'
```

Matrix room creation: easiest via Element UI. Or via Matrix API:

```bash
curl -X POST -H "Authorization: Bearer $MATRIX_TOKEN" -H "Content-Type: application/json" \
  "https://matrix.example.net/_matrix/client/v3/createRoom" \
  -d '{"preset":"private_chat","name":"NewApp","topic":"NewApp dev room","invite":["@claude:matrix.example.net"]}'
```

### 7. Smoke-test

Create a trivial `NEWAPP-9999` issue in YouTrack. Watch:
- Receiver webhook fires → Runner workflow
- Derive Slot output includes `cwd=/home/.../products/newapp`, `room=!NewAppRoom:...`
- Claude session lands in newapp cwd
- Lock file at `gateway-state/gateway.lock.newapp`
- Matrix message lands in #newapp

If all 5 work, you're onboarded.

## Day 1+: dispatch the first parallel-dev feature

### 8. Generate YouTrack epic + child tasks from tasks.json

```bash
bootstrap-pack/scripts/tasks-to-epic.py /app/products/<slug> 001-<feature-slug>
```

Creates one Epic in NEWAPP + one Task per row in tasks.json, all subtask-linked. Each child carries files_owned + acceptance_test + risk_score in its description.

### 9. Dispatch

Move the epic to `In Progress` in YouTrack. n8n Planner workflow fires, but since PROJECT.json exists, it skips Claude-driven decomposition and reads tasks.json directly. work_units rows land in SQLite. Distribute workflow fans out the wave-0 tasks.

### 10. Monitor + merge

Per [`parallel-dev-feature.md`](parallel-dev-feature.md). The merge-coordinator opens the MR; risk classifier flags it `[NEEDS-HUMAN]` if high-risk.

## Total clock time

| Phase | Time |
|---|---|
| /bootstrap session (interview + spec authoring) | 1-4 hours depending on project complexity |
| Phase F gate + iterate on failures | 30 min |
| project-onboard.sh + n8n Derive Slot edits | 30 min |
| YouTrack project + Matrix room setup | 10 min |
| Smoke-test | 10 min |
| **Total to dispatch-ready** | **~3-6 hours** |

After that, each new feature within the project is one `/specify → /tasks → dispatch` cycle — usually under an hour.

---

## When NOT to use this

- **Tiny fixes / single-file edits:** overkill. Just edit + commit + MR like normal.
- **Existing brownfield code:** bootstrap-pack is greenfield-oriented. For existing repos, just add a `PROJECT.json` + retroactively generate `spec/` for new features only.
- **Cross-repo features touching multiple sub-repos:** the worktree allocator only operates on ONE git repo at a time. Cross-cubeos-sub-repo features need single-agent dispatch.
- **Cost-sensitive iterations:** the bootstrap session itself burns 50k+ tokens. Don't /bootstrap a 200-LOC utility.

For the boundary cases above, single-agent ChatDevOps still works the same way it always has.
